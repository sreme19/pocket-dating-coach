# Requirements Document

## Introduction

This document defines the requirements for wrapping the existing Pocket Dating Coach SvelteKit application (deployed on Vercel) inside a native Android shell using Capacitor. The native app loads the live production URL directly, preserving all server-side logic while adding app store presence, push notifications, and device API access. The architecture supports future iOS extension without modification to shared configuration.

## Glossary

- **Android_Shell**: The native Android application generated by Capacitor that hosts the WebView and provides platform integration
- **WebView**: The system Chrome-based browser component embedded in the Android shell that renders the remote web application
- **Capacitor_Runtime**: The bridge layer that connects native device APIs to the web application running in the WebView
- **FCM**: Firebase Cloud Messaging — Google's push notification delivery service
- **Device_Token**: A unique string issued by FCM that identifies a specific app installation for push notification delivery
- **Keystore**: A Java KeyStore (.jks) file containing the cryptographic key pair used to sign release builds
- **AAB**: Android App Bundle — the required distribution format for Google Play Store submissions
- **Play_Store**: Google Play Store — the distribution channel for Android applications
- **Vercel_URL**: The production HTTPS URL where the SvelteKit application is deployed
- **Capacitor_Config**: The `capacitor.config.ts` file that defines app identity, server URL, and plugin settings
- **RLS**: Row Level Security — Supabase database policies that restrict data access to authorized users

## Requirements

### Requirement 1: Capacitor Integration

**User Story:** As a developer, I want to integrate Capacitor into the existing SvelteKit project, so that I can generate a native Android shell that loads the live Vercel deployment.

#### Acceptance Criteria

1. WHEN the Capacitor_Config is created, THE Capacitor_Runtime SHALL set `appId` to a reverse-domain identifier matching the pattern `com.<org>.<appname>` that remains unchanged across all subsequent builds of the application
2. WHEN the WebView initializes, THE Capacitor_Runtime SHALL load the production Vercel_URL specified in `server.url`, where the URL uses the HTTPS scheme and points to the deployed Vercel domain
3. THE Capacitor_Config SHALL set `server.cleartext` to `false` to enforce HTTPS-only communication and set `android.allowMixedContent` to `false` to prevent mixed HTTP/HTTPS resource loading
4. WHEN Capacitor is added to the project, THE Capacitor_Runtime SHALL not modify any existing SvelteKit source files, API routes, or adapter configuration; only new Capacitor-specific files and directories (capacitor.config.ts, android/) and `package.json` dependency entries shall be added
5. THE Capacitor_Config SHALL declare push notification presentation options with `badge`, `sound`, and `alert` each set to enabled
6. THE Capacitor_Config SHALL set `webDir` to the SvelteKit build output directory so that `npx cap sync` completes without file-not-found errors
7. IF the WebView cannot reach the configured `server.url` within 10 seconds, THEN THE Capacitor_Runtime SHALL display an offline error indication to the user rather than a blank screen

### Requirement 2: Native Android Shell

**User Story:** As a user, I want to install the app from the Play Store and use it like a native application, so that I get a familiar mobile experience with proper navigation and lifecycle handling.

#### Acceptance Criteria

1. THE Android_Shell SHALL declare INTERNET and POST_NOTIFICATIONS permissions in the AndroidManifest
2. THE Android_Shell SHALL set minimum SDK version to 22 (Android 5.1) and target SDK version to 34
3. WHEN the user presses the back button and the WebView has navigation history, THE Android_Shell SHALL navigate back one entry within the WebView history
4. IF the user presses the back button and the WebView has no navigation history, THEN THE Android_Shell SHALL exit the application
5. THE Android_Shell SHALL launch in portrait orientation with singleTask launch mode
6. WHEN the Android process is killed and recreated, THE Android_Shell SHALL restore the WebView to the last loaded URL persisted in instance state
7. IF the restored URL fails to load after process recreation, THEN THE Android_Shell SHALL fall back to loading the configured Vercel_URL
8. WHILE the WebView is loading the remote content, THE Android_Shell SHALL display a branded splash screen for no longer than 10 seconds before transitioning to an error or loaded state

### Requirement 3: WebView Content Loading

**User Story:** As a user, I want the app to load and display the full web application seamlessly, so that all existing features (auth, chat, profile review) work identically to the browser experience.

#### Acceptance Criteria

1. WHEN the app launches with network connectivity, THE WebView SHALL load the Vercel_URL and render the server-side rendered HTML within 30 seconds of launch
2. WHEN navigation is attempted to a URL outside the configured `server.url` origin, THE WebView SHALL block the in-app navigation and open the URL in the device's default system browser
3. WHEN the user interacts with the application, THE WebView SHALL forward all API calls to existing Vercel routes without modification
4. WHILE the WebView is loading remote content, THE Android_Shell SHALL display a visible loading indicator, and SHALL dismiss the indicator once the page has finished loading
5. IF the WebView encounters an SSL/TLS certificate error, THEN THE Android_Shell SHALL block navigation, prevent the user from bypassing the error, and display a security warning
6. THE WebView SHALL retain cookies, local storage, and session data between app launches so that user authentication state persists across sessions
7. IF the Vercel_URL fails to load within 30 seconds while network connectivity is available, THEN THE Android_Shell SHALL display an error state with a retry option

### Requirement 4: Push Notifications via FCM

**User Story:** As a user, I want to receive timely reminders about conversations and follow-up prompts, so that I stay engaged and don't miss opportunities.

#### Acceptance Criteria

1. WHEN the app launches for the first time after installation, THE Android_Shell SHALL request push notification permissions from the user via the system permission dialog
2. WHEN notification permissions are granted, THE Capacitor_Runtime SHALL register the device with FCM and obtain a Device_Token within 10 seconds of permission grant
3. WHEN a Device_Token is obtained and the user is authenticated, THE Android_Shell SHALL send the token to the backend API endpoint for storage associated with the authenticated user within 5 seconds
4. WHILE the app is in the foreground, WHEN a push notification is received, THE Android_Shell SHALL display an in-app notification banner showing the notification title, body, badge count, and sound
5. WHILE the app is in the background, WHEN a push notification is received, THE Android_Shell SHALL display a system notification in the device notification tray
6. WHEN the user taps a notification containing a deep link, THE Android_Shell SHALL navigate the WebView to the route specified in the `deepLink` data payload
7. WHEN the user taps a notification that does not contain a deep link, THE Android_Shell SHALL bring the app to the foreground and display the most recent screen
8. IF push notification registration fails, THEN THE Android_Shell SHALL continue functioning without notifications and re-attempt registration on the next app launch
9. IF the user denies push notification permissions, THEN THE Android_Shell SHALL continue functioning without notifications and shall not re-prompt the permission dialog until the next app install
10. WHEN the user logs out, THE Android_Shell SHALL delete the Device_Token from local storage and send a removal request to the backend API to disassociate the token from the user

### Requirement 5: Device Token Management

**User Story:** As a system operator, I want device tokens stored securely and associated with authenticated users, so that notifications reach the correct recipients.

#### Acceptance Criteria

1. THE Backend SHALL store Device_Token records with userId, token (maximum 256 characters), platform (limited to "android" or "ios"), and creation timestamp
2. THE Backend SHALL enforce RLS policies so that device tokens are readable, insertable, and deletable only by the owning user
3. WHEN the app launches, THE Android_Shell SHALL retrieve the current FCM token and send a registration request to the Backend, replacing any previously stored token for the same userId and platform combination
4. THE Backend SHALL validate that the userId in a token registration request corresponds to an authenticated Supabase session
5. IF a token registration request fails validation due to a mismatched or unauthenticated userId, THEN THE Backend SHALL reject the request and return an error response indicating an authentication failure
6. IF a notification send fails due to an invalid or expired FCM token, THEN THE Backend SHALL remove the invalid token from storage within the same operation
7. THE Backend SHALL store at most one Device_Token per userId and platform combination, replacing any existing token on new registration

### Requirement 6: Notification Payload Construction

**User Story:** As a developer, I want notification payloads to conform to FCM format and platform constraints, so that notifications display correctly on all Android devices.

#### Acceptance Criteria

1. THE Backend SHALL construct notification payloads containing a title of 1 to 65 characters and a body of 1 to 240 characters
2. IF a generated notification title exceeds 65 characters or body exceeds 240 characters, THEN THE Backend SHALL truncate the text to the maximum allowed length before sending
3. THE Backend SHALL include a `type` discriminator in the notification data payload with values limited to conversation_reminder, follow_up_prompt, or profile_tip
4. WHEN a notification includes a deep link, THE Backend SHALL include the target route in the data payload as `deepLink` formatted as a relative path beginning with `/`
5. THE Backend SHALL send notifications only to Device_Tokens that exist in storage and have not been flagged as invalid by a prior failed delivery attempt
6. THE Backend SHALL construct every notification payload with both the `notification` object (containing title and body) and the `data` object (containing `type` and, when applicable, `deepLink`)

### Requirement 7: Release Signing and Distribution

**User Story:** As a developer, I want a secure and repeatable release process, so that I can publish updates to the Play Store without risking keystore loss or signing inconsistencies.

#### Acceptance Criteria

1. THE Android_Shell SHALL be signed with a release keystore using the same key alias across all builds
2. THE Android_Shell SHALL produce an AAB (Android App Bundle) for Play Store submission
3. WHEN a release is submitted to the Play_Store, THE Android_Shell SHALL use a `versionCode` that is strictly greater than the previously submitted `versionCode` and a `versionName` following semantic versioning (MAJOR.MINOR.PATCH)
4. THE Keystore SHALL be stored outside version control with `.gitignore` entries for `*.jks` and `*.keystore` files
5. THE Keystore passwords SHALL be stored in a secrets manager and SHALL NOT appear in source code, build scripts, or version-controlled configuration files
6. WHEN a release is built, THE Android_Shell SHALL target the production signing configuration using the release keystore defined in criterion 1
7. IF the release build fails due to a missing keystore or incorrect signing credentials, THEN THE Android_Shell build process SHALL terminate with an error message indicating the signing failure cause without exposing password values
8. WHILE a debug build is being produced, THE Android_Shell SHALL use the default debug keystore and SHALL NOT reference the release keystore or its credentials

### Requirement 8: Play Store Listing

**User Story:** As a product owner, I want a complete Play Store listing, so that the app passes review and presents well to potential users.

#### Acceptance Criteria

1. THE Play_Store listing SHALL include a title (max 30 characters), short description (max 80 characters), and full description (max 4000 characters), each containing at least 2 characters
2. THE Play_Store listing SHALL include at minimum 2 and at most 8 screenshots for phone form factor, each with dimensions between 320px and 3840px per side in PNG or JPEG format
3. THE Play_Store listing SHALL include a 1024x500 feature graphic in PNG or JPEG format and a 512x512 app icon in PNG format with no transparency
4. THE Play_Store listing SHALL include a privacy policy URL that is publicly accessible over HTTPS without requiring authentication
5. THE Play_Store listing SHALL include an IARC content rating questionnaire result
6. THE Play_Store listing SHALL categorize the app under the "Lifestyle" category in the Google Play Store
7. THE Play_Store listing SHALL declare the app as free with no in-app purchases and select a target audience age group of 18 and above

### Requirement 9: Offline Graceful Degradation

**User Story:** As a user, I want the app to handle network loss gracefully, so that I understand what's happening and can resume when connectivity returns.

#### Acceptance Criteria

1. IF the device has no network connectivity on launch, THEN THE Android_Shell SHALL display a full-screen offline state containing a visible offline icon, a text message indicating no internet connection, and a manual retry button
2. WHEN network connectivity is restored after an offline state, THE Android_Shell SHALL automatically retry loading the Vercel_URL within 3 seconds of detecting the connectivity change
3. WHILE the device is offline, THE Android_Shell SHALL not crash or produce unhandled exceptions and SHALL retain the last successfully loaded WebView content visible on screen
4. IF a network request fails during active use, THEN THE WebView SHALL display an inline error indicator containing a message indicating the connection was lost and a manual retry option
5. IF automatic retry after connectivity restoration fails, THEN THE Android_Shell SHALL reattempt loading a maximum of 3 times with a 5-second interval between attempts before displaying the offline state with a manual retry button

### Requirement 10: iOS Extensibility

**User Story:** As a developer, I want the architecture to support iOS without requiring changes to shared configuration, so that adding an iOS app is a minimal-effort extension.

#### Acceptance Criteria

1. THE Capacitor_Config SHALL be platform-agnostic, requiring no modifications when adding iOS support
2. WHEN iOS platform support is added, THE existing Android_Shell code and configuration SHALL remain unchanged
3. THE push notification registration logic SHALL support both android and ios platform identifiers in the Device_Token payload

### Requirement 11: Security

**User Story:** As a security-conscious developer, I want all communication encrypted and all secrets kept server-side, so that user data and API keys are protected.

#### Acceptance Criteria

1. THE WebView SHALL communicate exclusively over HTTPS with no HTTP fallback permitted, and SHALL block loading of mixed HTTP sub-resources (scripts, images, stylesheets)
2. THE Android_Shell SHALL not contain any server-side API keys (Anthropic, Supabase service role, Voyage, fal.ai) in source code, compiled bytecode, bundled assets, or string resources within the distributed AAB
3. WHEN the WebView detects a navigation request to a domain other than the configured Vercel_URL origin, THE Android_Shell SHALL block the navigation and keep the WebView on the current page
4. THE Backend SHALL store FCM Device_Tokens with RLS policies scoped to the owning user such that SELECT, INSERT, UPDATE, and DELETE operations are restricted to rows matching the authenticated user's ID
5. WHEN the user logs out, THE Android_Shell SHALL clear WebView cookies, local storage, session storage, and any cached authentication tokens, and SHALL send a request to the backend to delete the Device_Token record associated with the user and device
