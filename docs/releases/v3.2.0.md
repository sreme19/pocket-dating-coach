# v3.2.0 — Multi-Agent Chat, Server-Side Persistence & Public-Profile Parity

**Release date:** 2026-05-30
**Previous release:** v3.1.0
**Scope:** 51 commits · 49 files · +3,721 / −672

This release turns the AI multi-agent system (Matchmaker → Wingman → Bestie) into
a real, always-on experience and moves profile media + AI state off the browser
and onto the server. It also brings the public/Discover profile view to full
parity with the owner's profile, makes the whole app gender-aware, and fixes a
class of "works on my device only" bugs caused by localStorage-only state.

> ⚠️ **This release requires database migrations and one manual data step.**
> See [Database migrations](#database-migrations-run-in-order) and
> [Manual data operations](#manual-data-operations) before/at deploy.

---

## 1. AI Bestie — now autonomous, server-side, and persistent

The Bestie is the woman's private advisor that can also jump into a match
conversation and reply on her behalf. Previously it only worked while her browser
tab was open. It is now a first-class server-side agent.

- **Server-side auto-responder (the headline fix).** When a match messages a
  woman who has Bestie active, the reply is now generated and sent **server-side
  inside the `chat/send` request** — so it works even when her app is closed or
  backgrounded. New shared lib: `src/lib/server/bestie-responder.ts`
  (`generateBestieReply` / `generateAndSendBestieReply`) loads her preferences,
  the match's verified artifacts, and recent conversation history, calls Claude,
  and returns `{ signal, read, suggestedQuestion }`.
- **Client no longer sends.** The old client-side poller/realtime auto-send
  (`generateAndSendAIBestieResponse` + `respondedToMessageIds` bookkeeping) was
  removed. The client only fetches and displays now — no double-responses.
- **On by default.** `verified_vibe_matches.ai_bestie_active` now defaults to
  `TRUE`; existing rows backfilled. Bestie persists from the DB (source of
  truth), not just localStorage.
- **Represents her, never impersonates her.** The Bestie speaks in its own voice
  as her bestie — referring to her in the third person by name — and is honest
  about who it is. If the man asks "who am I talking to?" it answers plainly
  ("you're talking to her AI bestie, helping her get to know matches first"). It
  never claims to be her or pretends a message came from her. The single source
  of truth for this persona is `buildBestieReplyPrompt` in `src/lib/prompts.ts`;
  every match-facing endpoint builds its prompt there so the persona can't drift.
- **Uses real context.** The responder receives the last ~12 messages, so it
  stops repeating questions or contradicting what was already said, and treats
  the woman's own typed messages as part of one continuous voice.
- **Steps back when she takes over.** If the woman has manually replied after the
  match's latest message, Bestie stays silent.
- **Correct identity everywhere.** Replaced the hardcoded "Neha" with the actual
  user's first name across the Bestie chat and generate-response prompts.
- **Auto-named persona.** On first setup the Bestie is assigned a random common
  name (Priya, Aisha, Maya, …) and it's persisted; the owner can rename it.

### Coaching cards
- The Bestie's private read (`signal` ✅/⚠️/🚩 + one-line `read`) is stored on the
  triggering message via new `ai_signal` / `ai_read` columns, returned by the
  conversation API, and rendered under the man's message — **visible to the woman
  only**.

### Visual distinction in chat
- AI-sent messages render with a **purple→pink gradient** bubble (vs the user's
  green), backed by a persisted `is_ai` flag so both sides see it consistently.
- Every message now shows a **WhatsApp-style avatar + sender name**. AI messages
  are labelled **"<Owner>'s AI Bestie"** with a ✨ gradient avatar.

### "Bestie's Take" profile flags (female viewers, male profiles)
- New endpoint `POST /api/verified-vibe/bestie-profile-flags` analyses a man's
  verified proofs against his archetype claims and surfaces 2–4 orange/red flags
  (e.g. "globe-trotter" archetype but only one verified destination). Rendered as
  a card on both the standalone public profile and the Discover profile view.

## 2. AI Wingman & compliance

- **Context-aware compliance gate.** `complianceGate` now takes a
  `context: 'advisor' | 'outbound'`. The private advisor chats (Bestie, Wingman,
  greetings) use a relaxed validator that only blocks genuine harms (phone /
  email / address / government-ID / financial details / explicit content /
  medical-legal advice / outcome guarantees) while **allowing** candid discussion
  of the user's own matches. This fixes advisor replies being silently swapped
  for the canned "I want to make sure I'm being as helpful as possible…"
  fallback. Outbound messages to other users keep the strict validator.

---

## 3. Multi-agent privacy & loop safety

- **No Bestie↔Wingman feedback loop.** Bestie-sent messages are tagged and never
  treated as new inbound messages; the responder only fires on genuine,
  non-AI messages addressed to the woman.
- **Strict gender gating.** Bestie coaching cards, auto-responses, and the intro
  card are fully gated to female users — the male side never runs Bestie or sees
  its cards.

## 4. Public profile & Discover parity

- **Single shared body.** New `PublicProfileBody.svelte` renders the public read;
  the standalone profile page and the Discover sheet now render **identically**.
- The public read now mirrors the owner's rich profile — all sections, Money
  Matters, AI Portrait, Travel Magnets, Garage, verified signals.
- Discover card taps navigate to the exact public-read view.
- `match-profile` API returns the live trust score and a title-cased name.

## 5. Gender-aware experience

- Discovery, onboarding, AI greetings, and the AI-portrait gate are now
  gender-aware end to end.
- Gender is persisted to `user_metadata` at OTP send, with correct fallbacks for
  users that don't yet have a profile row.
- Female profiles no longer show male-only affordances ("Enhance with AI",
  "AI will enhance" photo hints).
- Archetype is a **ranking signal, not a hard filter** in the discovery feed, and
  real profiles are shown to unverified users instead of bailing to seed data.

## 6. Server-side media persistence (no more localStorage-only)

The root cause of "the photo shows on my device but not my match's": profile
photos lived only as base64 in `localStorage`.

- **Photos upload to Supabase Storage.** New `POST /upload-photo` and
  `POST /upload-avatar` write to the public `profiles` bucket
  (`photos/{userId}/{label}.{ext}`, `avatars/{userId}/lead.{ext}`) and return a
  CDN URL. The lead photo also mirrors to `avatar_url`.
- **Canonical list on the server.** `photos[]` / `aiPhotos[]` (hosted URLs +
  labels) persist in `user_master_profile.data` and hydrate on any device.
- **One-time migration on load** uploads any legacy base64 photos still in
  localStorage and replaces them with hosted URLs.
- **Service worker fix.** Bumped caches `v1 → v2` and switched Supabase Storage
  object URLs to **network-first**, so an updated avatar shows for matches
  immediately instead of serving a stale cached app shell/image.

---

## 7. Onboarding, verification & access fixes

- **Real users no longer stuck in verification.** `is_seed = false` users are
  grandfathered past the verification gate and routed straight to Discover;
  `upsertProfile` now stamps `is_seed = false` on every real signup.
- **Clear-chat for everyone.** Removed the seed-only restriction on the
  clear-conversation button and its endpoint.
- **Rate limit raised** from 20 → 500 requests/hr per IP (20 was hit during
  normal chat usage: load + poll + mark-read + send all count).

---

## 8. Admin / analytics

- New admin analytics route (`/admin/analytics`) and
  `GET /api/analytics/user-activity` for user-activity reporting (seed vs real).

All are idempotent (`IF NOT EXISTS` / guarded). Apply against the `Woam` project
(`stikoktiaxqtcsohcxzp`).

1. `20260530_is_seed_flag.sql` — add `is_seed` to `verified_vibe_users`; mark known real users.

4. `20260530_bestie_active_by_default.sql` — `ai_bestie_active` default `TRUE` + backfill existing matches.
5. `20260530_messages_is_ai.sql` — add `is_ai` to `verified_vibe_messages` (AI-message styling, both sides).

> The send path is defensive: it omits `is_ai` from inserts until the column

## Configuration / behavior changes to be aware of

- **API rate limit:** 500 req/hr/IP (was 20).
- **Service worker cache version:** v2 — clients drop v1 caches on next load.
- **AI Bestie default:** ON for all matches.
- **Avatar storage:** `avatar_url` now holds a public Storage CDN URL, not base64.
  Existing base64 avatars are migrated when the owner next opens their profile.
